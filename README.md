# CRA Evidence OS

[Live application](https://cra-evidence-os-build-week.perapericmika.chatgpt.site) · [Public demo video](https://youtu.be/pRDlUOtt7GE)

OpenAI Build Week **Work & Productivity** entry: a traceable evidence workflow for EU Cyber Resilience Act preparation. It turns synthetic product documents and SBOMs into grounded GPT-5.6 suggestions, explicit human approvals, a change-aware Evidence Graph, incident worksheets, and a draft Annex VII package.

> Evidence readiness is not legal compliance. AI never approves claims, decides stale state, or submits incidents.

## Build Week slice

This repository started from commit `e46828b` with only a placeholder README. The contest slice is the complete application history after that point. The post-contest roadmap is intentionally not presented as implemented.

## Run locally

Requires Node.js 22+.

```bash
npm install
copy .env.example .env.local
npm run dev
```

Real model calls require a server-side `OPENAI_API_KEY`. The configured model is `gpt-5.6-sol`, using the Responses API, strict Structured Outputs, reasoning effort `medium`, and `store: false`. No API key means an explicit error; production never substitutes a fixture result.

## Server-side AI routes

- `POST /api/v1/ai/evidence` — extracts one schema-valid suggestion from a known source fragment and persists it as `SUGGESTED`.
- `POST /api/v1/ai/approve` — records a separate authenticated reviewer decision and changes the persisted suggestion to `APPROVED`.
- `POST /api/v1/ai/impact` — explains a deterministic SBOM blast radius without changing the code-selected `STALE` set.
- `POST /api/v1/ai/incident` — drafts a worksheet only from approved facts and never submits it.
- `GET /api/v1/ai/health` — reports only whether server-side configuration exists, plus model and reasoning metadata; it never returns a secret value.

Mutation inputs reject unknown fields and unknown source/case identifiers. Identity comes from the authenticated Sites request, tenant scope is derived server-side, and model calls are limited per authenticated user, tenant, and use case to six requests per ten-minute window.

```bash
npm run lint
npm run typecheck
npm test
npm run test:integration
npm run test:ai-evals
npm run test:ai-evals:live
npm run test:e2e
npm run build
npm run demo:reset
```

The integration test mocks the OpenAI HTTP response and checks model/reasoning/schema configuration, safe metadata, a valid source fragment, and absence of sensitive response fields. The live smoke test uses synthetic data and fails closed when a server-side API key is unavailable.

## GPT-5.6 contribution

1. Extract grounded claim suggestions with source fragment IDs.
2. Explain deterministic SBOM impact and prioritize review.
3. Draft 24h/72h incident fields only from approved facts while preserving unknowns.

Every model output is locally schema-validated and remains `SUGGESTED`; application code, not the model, determines stale status. See `docs/CODEX_AND_GPT56.md` and `docs/GPT56_EVAL_REPORT.md`.

The Evidence Inbox accepts PDF or JSON up to 5 MB, verifies file signatures, hashes bytes, stores blobs in R2, and stores tenant-scoped metadata and audit events in D1. The Dossier screen downloads a physical PDF and ZIP manifest package generated by `scripts/build-demo-export.py`.

## Security boundaries

Synthetic data only. Documents are untrusted input. Tenant scope is server-derived. No arbitrary URL fetch, code execution, automatic ENISA submission, secret reflection, Authorization reflection, complete-prompt reflection, or hidden live-AI fallback. Production storage uses D1 and R2. Configure a project-level OpenAI spend limit in addition to the application rate limit.
